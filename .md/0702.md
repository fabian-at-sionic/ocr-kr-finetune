# 2026-07-02 Revised Training Strategy And Failure Analysis

This memo supersedes the earlier scale-up strategy. The project is still SOTA-oriented Korean/Hangul detection OCR, but the training process must now be checkpoint-gated. The Stage 1 50K LoRA failure showed that finishing a long run is not evidence of progress.

## What Failed

The completed Stage 1 50K LoRA adapter:

```text
runs/train/stage1_lora_50k_balanced_textonly_v3/final_adapter
```

failed badly on the KORIE receipt-field OCR benchmark:

```text
Exact match: 0.00%
CER: 2529.08%
WER: 345.65%
```

Qualitative output was not normal OCR error. It was repeated junk generation such as:

```text
amotoamotoamoto...
iginaliginal...
```

The adapter also generated multimodal/control special tokens during initial inference attempts, requiring token suppression in the benchmark runner. That is a model-output stability failure, not just a weak CER score.

## Confirmed Causes

These are facts from the run artifacts and benchmark report:

1. **The final 50K adapter was selected by completion, not validation.**
   The run finished, but there was no required evaluation gate before treating the adapter as useful.

2. **KORIE exposed repeated-generation collapse.**
   The predictions often ran to generation limits with repeated fragments. This inflated edit distance and produced catastrophic CER/WER.

3. **The 50K adapter generated control/image tokens unless suppressed.**
   The benchmark initially hit Granite/Docling image-token mismatch errors after the adapter generated tokens such as `<image>` / fake image tokens. Suppressing those tokens made inference finish, but the need for suppression is itself a failure signal.

4. **KORIE differs from the Stage 1 training distribution.**
   Stage 1 used broad OCR recognition data. KORIE is cropped Korean receipt fields with many numbers, dates, merchant names, phone numbers, and tiny low-resolution fields.

5. **The 50K run had warning signs in the loss tail.**
   The KORIE failure report notes abnormal late-run loss spikes above 140. That should have triggered checkpoint comparison before using the final adapter.

## Likely Contributing Factors

These are plausible but should be verified by checkpoint evaluation:

- The final checkpoint may be worse than an earlier checkpoint such as `checkpoint-3000`.
- Learning rate / run length may have been too aggressive for the Stage 1 mixture.
- The generic prompt may be too weak for KORIE receipt-field crops.
- The benchmark decoding was not strict enough to stop repetition early.
- The model may have overfit response style without learning robust short-field OCR.
- Text-only LoRA may have shifted generation behavior enough to destabilize special-token handling.

## Revised Non-Negotiable Policy

No future long run is accepted because it completed.

A checkpoint is promoted only if it passes bounded evaluation:

```text
train/checkpoint -> evaluate -> promote or reject -> continue only from accepted checkpoint
```

Required gate metrics:

- CER
- WER
- exact match
- blank rate
- repetition-loop rate
- generated/target length ratio
- control-token generation count
- JSON/schema validity for structured tasks
- sample predictions

Reject a checkpoint if:

- repeated fragments appear frequently,
- generated/target length ratio is extreme, for example >3x,
- KORIE CER is more than 2x the current accepted checkpoint,
- AIHub91 validation CER worsens substantially for two consecutive checkpoints,
- control/image tokens appear in generated text,
- outputs run to max token length instead of stopping naturally.

## 2026-07-01 23:43 UTC - Repeated LoRA Rejection Diagnosis And Fix Plan

The repeated checkpoint rejections are best classified as a generation-format failure, not only weak OCR accuracy.

Evidence:

- Stage 1 checkpoint-3000 and Stage 1 final_adapter generated repeated fragments such as `amoto` and `iginal` on both AIHub91 region crops and KORIE crops.
- Stage 2A checkpoint-200 generated role/control-like text such as `assistant`, `user`, and repeated control-looking fragments.
- Multiple checkpoints emitted or attempted to emit image/control/doc tokens, requiring generation-time token suppression to prevent Idefics3 image-token mask errors.
- Some Stage 2A checkpoints blanked out instead of producing OCR text.
- Output length was often extreme: examples include generated/target length ratios above 40x and 80x.

This means the model is not merely misreading Korean text. It is failing to stay inside the expected answer format. More rows from the same setup can amplify the collapse.

### Likely Causes

1. Training targets may still contain prompt/template/docling/control artifacts, or the loss may include tokens outside the intended OCR answer span.
2. The base model is a document-to-docling model, while Stage 1 and Stage 2A are asking for plain OCR strings. That mismatch can push the model toward doc tags, role tokens, or generic generation artifacts unless the target format is tightly constrained.
3. Some crops are very small or visually ambiguous. If the answer distribution includes many short strings, blank strings, or noisy labels, the adapter can learn unstable shortcuts.
4. The Stage 1 50K adapter appears already collapsed; Stage 2A from that parent cannot be trusted to recover it.
5. Evaluation currently proves that generation suppression can make runs complete, but needing suppression is itself a warning sign.

### Resolution Plan

Do not resume long Stage 2A from any rejected parent. Rebuild the training path with a format gate before scaling.

1. Establish clean baselines first.
   - Run IBM Granite/Docling 258M base on the same KORIE split.
   - Run the same base on the fixed AIHub91 printed region validation sample.
   - Compare against every adapter using the same prompt, crop preprocessing, max tokens, and suppression policy.

2. Audit training examples before any new run.
   - Sample at least 1,000 manifest rows.
   - Verify target strings contain only intended OCR text.
   - Reject rows containing `<image>`, `<fake_token_around_image>`, `<|start_of_role|>`, `<|end_of_role|>`, `assistant`, `user`, docling tags, or serialized chat text.
   - Cap target length by task type and inspect outliers.

3. Mask loss to answer tokens only.
   - The model should not learn from user prompt tokens, assistant role tokens, image placeholder tokens, or chat template delimiters.
   - Only the final OCR answer span should contribute to loss.

4. Make the output contract explicit and short.
   - For region OCR, train and evaluate with a plain-text target only.
   - Keep `max_new_tokens` close to the expected target length plus margin.
   - For AIHub91 word/region OCR, prefer a low cap such as 32 or 64 unless validation proves longer is needed.

5. Add hard generation gates during validation.
   - Reject any checkpoint that emits control/image/role tokens.
   - Reject repetition loops even if CER appears superficially better.
   - Track blank rate, average generated length, and generated/target ratio.

6. Restart from base or the best non-collapsed checkpoint only.
   - Current rejected Stage 1 final_adapter and checkpoint-3000 should not be used as parents.
   - If a Stage 1 adapter is retrained, run a small 1K-5K smoke first, evaluate, then scale only after the gate passes.

7. Use lower-risk LoRA settings for the next smoke.
   - Lower learning rate than the collapsed Stage 1 run.
   - Smaller number of trainable modules if possible.
   - Conservative max answer length.
   - Save frequently and evaluate every checkpoint before continuing.
